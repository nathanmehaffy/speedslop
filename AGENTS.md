# Implicit Evolutionary Simulation: Agent Instructions

This repo contains a high-performance simulation of agents controlled by small neural networks. The aim of this repo is to create a simulation in which complex behavior can evolve and emerge without hardcoding the evolutionary rules. Instead, breeding and death are available pathways, and evolution emerges naturally (or that is the aim).

To avoid extinction, the population must be held at constant size, and all interaction rules must maintain this invariant.

This is a new/greenfield project, and as such backwards compatibility, fallbacks, and other development safety paranoia are never wanted or needed. The project should always be maintained as a small, clean codebase that does exactly what it needs to do, and does not have anything extra left over from old versions that is no longer needed.

All code pathways should be extensively tested, especially the simulation core.

## Onboarding Summary

SpeedSlop is a browser-based evolutionary simulation with a Rust simulation core compiled to WebAssembly and a TypeScript/WebGPU frontend. The simulation state lives in Rust; the browser only drives time, uploads a packed render buffer, displays HUD controls, and draws the agents.

## Repository Structure

- `sim/`: Rust crate for the simulation core, exported to JavaScript with `wasm-bindgen`.
- `sim/src/lib.rs`: The main simulation implementation and Rust unit tests.
- `src/main.ts`: Browser entrypoint. Initializes WASM, creates the `Simulation`, sets up WebGPU, handles pause/reset/speed controls, and runs the animation loop.
- `src/style.css`: Fullscreen canvas and compact HUD styling.
- `src/wasm-pack.d.ts`: Local TypeScript declarations for the generated `sim/pkg/speedslop.js` module.
- `index.html`: Canvas, HUD metrics, controls, and Vite script entry.
- `package.json`: Build, dev, and check scripts.
- `sim/pkg/`: Generated by `wasm-pack build sim --target web`; imported by `src/main.ts`.
- `dist/`: Generated production frontend bundle.

## Runtime Architecture

The Rust `Simulation` owns all authoritative state:

- positions, directions, speed, color, age, and generation
- neural-network genomes and temporary network outputs
- breeding state, death flags, and spatial hash grid
- a stable packed `render_agents` buffer exposed to JavaScript

The TypeScript app imports `Simulation` from the generated WASM package, calls `tick(dt)` each animation frame, creates a `Float32Array` view into WASM memory using `agent_ptr()` and `agent_f32_len()`, and uploads that data to a WebGPU storage buffer.

WebGPU renders each agent as one instanced triangle. The shader expects each agent to occupy eight `f32` values:

```text
[x_norm, y_norm, dir_x, dir_y, r, g, b, speed_norm]
```

Because the GPU shader treats the packed buffer as pairs of `vec4f`, any change to `RENDER_STRIDE_FLOATS` or the render-buffer layout in Rust must be mirrored in `src/main.ts`.

## Simulation Core

The world is toroidal: positions wrap at the world bounds, and local distance checks use wrapped deltas. The spatial grid uses fixed-size cells to keep vision, collision, and mating checks local.

Each fixed simulation step currently does this:

1. Rebuild the spatial grid.
2. Write sensory inputs and evaluate each agent's neural network.
3. Apply turn, acceleration, movement, color, and age updates.
4. Rebuild the grid.
5. Resolve lethal side/body collisions by immediately replacing dead agents with randomized agents.
6. Rebuild the grid.
7. Resolve mating. Successful breeding replaces one parent with a mutated child genome.

The constant-population invariant is central. Death is implemented as replacement, and birth is implemented as replacing an existing agent, so `population()` must remain unchanged across all interactions.

## Neural Agents

Agents use a small one-hidden-layer neural network:

- `INPUT_COUNT`: sensory and self-state values.
- `HIDDEN_COUNT`: hidden layer width.
- `OUTPUT_COUNT`: turn, acceleration, and RGB color targets.
- `GENOME_LEN`: derived from the network shape and tested in Rust.

Vision is ray-based. Each ray reports whether another agent was seen, normalized distance, and observed RGB color. Self inputs include normalized speed and capped age.

## Development Commands

Run from the repository root:

```powershell
npm install
npm run dev
npm run build
npm run check
cargo test --manifest-path sim/Cargo.toml
```

`npm run dev` builds the WASM package first and then starts Vite on `127.0.0.1`. `npm run build` builds WASM, type-checks TypeScript, and creates the production bundle. `npm run check` builds WASM, type-checks TypeScript, and checks the Rust crate for the `wasm32-unknown-unknown` target.

## Testing Expectations

Keep the simulation core heavily tested in Rust. Tests should cover invariants and behavior, especially:

- population remains constant after ticks, deaths, and breeding
- toroidal wrapping and wrapped distance logic
- deterministic seeded randomness
- genome/network shape contracts
- mutation bounds
- collision rules
- breeding eligibility and replacement semantics
- render-buffer length, stride, and value ranges

Frontend changes should at least pass `npm run build`; simulation changes should also pass `cargo test --manifest-path sim/Cargo.toml`.

Do not use the Codex browser tool for browser testing in this repo. The developer will run all browser testing manually in his own browser because that workflow is faster.

## Engineering Preferences

This is a young project, so prefer direct, small, readable code over compatibility layers or defensive leftovers from previous versions. When changing architecture, remove obsolete paths instead of preserving unused alternatives.

Keep high-performance data flow in mind. The Rust core uses structure-of-arrays storage and exposes a packed render buffer to avoid per-agent JavaScript objects. Preserve that style unless there is a strong reason to change it.
